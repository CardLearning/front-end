# Card Learning: обзор проекта

## Краткое описание

Card Learning — приложение для запоминания новой информации с помощью карточек формата "вопрос / ответ". Пользователь создает наборы карточек, учит их короткими подходами и самостоятельно управляет повторениями: выбирает количество карточек за подход, длительность занятий и частоту возвращения к материалу.

На первом этапе приложение фокусируется на ручном интервальном повторении без встроенного алгоритма. Система помогает организовать карточки и процесс обучения, но решение о том, когда и сколько повторять, остается за пользователем.

## Цель продукта

Помочь пользователям регулярно запоминать слова, определения, факты и другие короткие единицы информации через простой и понятный процесс:

- создать набор карточек;
- пройти учебный подход;
- проверить себя по вопросу;
- открыть ответ;
- решить, когда вернуться к повторению;
- при необходимости поделиться набором с другими пользователями.

## Для кого приложение

Основная аудитория:

- люди, изучающие иностранные слова;
- студенты и школьники, запоминающие определения, даты, формулы и термины;
- специалисты, которые хотят закреплять профессиональные знания;
- пользователи, которым нужен простой инструмент без сложной настройки алгоритмов.

## Frontend-стек

Frontend-часть приложения разрабатывается на Next.js с использованием TypeScript.

Next.js выбран как основной frontend-фреймворк проекта. Все новые frontend-решения должны проектироваться с учетом возможностей Next.js, маршрутизации приложения и типизированной разработки на TypeScript.

Подробное frontend-решение описано в `docs/architecture/frontend.md`.

## Методология разработки frontend

Для frontend используется методология Frontend-first development (FFD).

Сначала разрабатывается пользовательский интерфейс на моковых данных, а затем backend реализуется под уже готовый интерфейс и согласованные API-контракты. Такой подход позволяет раньше проверить пользовательские сценарии, структуру экранов и состав данных, не блокируя frontend-разработку ожиданием backend.

Процесс разработки:

1. Проектирование UI.
2. Создание TypeScript типов.
3. Создание fixtures/mock data.
4. Разработка frontend.
5. Определение API-контрактов.
6. Реализация backend.
7. Замена fixtures на реальные запросы.

На ранних этапах frontend должен быть работоспособен на fixtures. Реальные API-запросы подключаются после того, как интерфейс, типы данных и контракты достаточно стабилизированы.

Правила формирования API-контрактов описаны в `docs/architecture/api-contracts.md`.

## Базовый функционал MVP

### Обучение по карточкам

Пользователь проходит карточки в формате "вопрос / ответ":

1. Видит вопрос.
2. Вспоминает ответ самостоятельно.
3. Открывает ответ.
4. Переходит к следующей карточке.

Карточка должна быть простой: вопрос на одной стороне, ответ на другой. На MVP не закладываем сложные типы карточек, вложения, тесты с вариантами ответа или автоматическую оценку правильности.

### Ручное интервальное повторение

Интервальное повторение используется как методика, но без обязательного алгоритма на первом этапе.

Пользователь сам определяет:

- сколько карточек брать в один подход;
- сколько раз повторять материал;
- как часто возвращаться к набору;
- когда считать карточку достаточно хорошо выученной.

Приложение может показывать базовую информацию о прогрессе, но не должно автоматически навязывать расписание повторений в MVP.

### Наборы карточек

Пользователь может создавать наборы карточек для разных тем.

Минимальная структура набора:

- название;
- описание;
- список карточек;
- автор;
- статус доступа.

Минимальная структура карточки:

- вопрос;
- ответ;
- принадлежность к набору;
- дата создания;
- дата последнего изменения.

### Шаринг наборов

Пользователь может делиться созданными наборами с другими пользователями.

На MVP достаточно заложить базовую модель доступа:

- приватный набор доступен только автору;
- расшаренный набор доступен другим пользователям по ссылке или через публичную страницу набора;
- пользователь, получивший доступ, может использовать набор для обучения.

Правила копирования, редактирования чужих наборов и совместной работы можно уточнить отдельно перед реализацией.

## Основные пользовательские сценарии

### Создание набора

1. Пользователь создает новый набор.
2. Указывает название и описание.
3. Добавляет карточки с вопросами и ответами.
4. Сохраняет набор.
5. Начинает обучение или делится набором.

### Обучающий подход

1. Пользователь выбирает набор.
2. Выбирает количество карточек на подход.
3. Запускает обучение.
4. По очереди проходит карточки.
5. Завершает подход и видит базовый результат.

### Повторение материала

1. Пользователь возвращается к набору через выбранный им интервал.
2. Запускает новый подход.
3. Повторяет карточки.
4. Самостоятельно решает, нужно ли продолжать обучение по набору.

### Использование чужого набора

1. Пользователь открывает расшаренный набор.
2. Просматривает описание и состав карточек.
3. Начинает обучение по этому набору.
4. При необходимости сохраняет набор себе, если такая возможность будет реализована.

## Дополнительный функционал после MVP

Эти функции важны для развития продукта, но не являются обязательными для первого рабочего релиза.

### Аудио воспроизведение

Возможные сценарии:

- озвучивание вопроса или ответа;
- воспроизведение произношения слов;
- загрузка пользовательского аудио;
- генерация аудио через внешний сервис.

Особенно полезно для изучения иностранных языков.

### Алгоритмы обучения

После ручного режима можно добавить автоматические алгоритмы:

- расчет даты следующего повторения;
- уровни знания карточки;
- автоматическое уменьшение или увеличение интервала;
- приоритизация сложных карточек.

Алгоритм должен быть отдельным слоем, чтобы ручной режим не ломался и мог оставаться доступным.

### Уведомления

Возможные каналы:

- email;
- Telegram.

Уведомления могут напоминать о повторении наборов, возвращении к сложным карточкам или поддержании серии занятий.

### Ассоциативные алгоритмы обучения

Можно развивать обучение через ассоциации:

- подсказки;
- мнемонические связи;
- группировку карточек по смыслу;
- визуальные или текстовые ассоциации;
- пользовательские заметки к карточкам.

Этот функционал стоит проектировать отдельно, потому что он может изменить структуру карточки.

### Ачивки и мотивация

Возможные достижения:

- количество дней занятий подряд;
- количество выученных карточек;
- количество завершенных подходов;
- создание определенного числа наборов;
- прохождение публичного набора.

Ачивки должны поддерживать регулярность, но не усложнять базовый процесс обучения.

### Мобильное приложение

Мобильная версия важна для обучения короткими подходами в течение дня.

Возможные варианты развития:

- адаптивная веб-версия;
- PWA;
- отдельное мобильное приложение.

На раннем этапе стоит проектировать интерфейс так, чтобы основные сценарии удобно работали на мобильном экране.

## Принципы продукта

- Простота важнее количества функций.
- Карточка должна создаваться быстро.
- Обучающий подход должен запускаться за несколько действий.
- Пользователь сохраняет контроль над процессом повторения.
- Расширенные алгоритмы не должны мешать ручному режиму.
- Шаринг наборов должен помогать обмену знаниями, а не усложнять владение контентом.

## Что не входит в MVP

В первый релиз не включаем:

- автоматический алгоритм интервального повторения;
- аудио;
- уведомления;
- ачивки;
- мобильное приложение как отдельный клиент;
- сложные типы карточек;
- совместное редактирование наборов;
- маркетплейс или рейтинг публичных наборов.

## Открытые вопросы

- Нужно ли пользователю копировать чужой набор себе или достаточно учиться по оригиналу?
- Должны ли карточки иметь статус изучения уже в MVP?
- Нужна ли регистрация для просмотра публичных наборов?
- Какие роли пользователей нужны на старте: только обычный пользователь или также администратор?
- Какой минимальный результат показывать после подхода: количество пройденных карточек, самооценку, время обучения или все сразу?
- Нужно ли хранить историю подходов в MVP?
